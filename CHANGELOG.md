## Type system (Track A)

- **A0 Type-algebra inventory** — `Alias`, `Function`, `Object` (structural + index sigs), literal types, `Union`, `Nullable`, `Intersection`, `Negation`, `Unknown`/`Infer`, `Never`, `Type`, `Custom`.
- **A3 Intersection surface syntax** — `A & B` parses, binds tighter than `|`, folds through `intersectType` at parse time; narrowing algebra (`intersectType`) done. Object-type structural intersection: `{a} & {b}` now merges field-wise to `{a, b}` (shared fields refine to their per-field meet; an impossible field → `Never`).
- **A3 fix: `never`-typed value rejected cleanly (no codegen crash)** — forming a VALUE for a binding/annotation whose type folds to `never` (`type C = A & B` with `A` an object and `B` a disjoint string literal) is now a clear, source-located type error — `Type \`C\` is \`never\` (an empty type — \`A & B\`, an object and a string literal, have no common values), so it has no values` — instead of slipping through to codegen and dying as the opaque `Unhandled AST -> WASM "Never" type`. The diagnostic RANGE points at the `A & B` intersection EXPRESSION (a source span the parser captures when it folds the `&` and threads alongside the empty-intersection tag), not the alias name or the value site. `ensureType`'s never branch reports when the TARGET is a (tagged) empty intersection, a real value flows into it, and a real `ctx` is present; legitimate `never` (a narrowed unreachable branch, a membership/narrowing query) is untouched. A defensive backstop in `toWasm.ts`'s `toWasmType` throws a clear compiler diagnostic if a `Never` ever reaches codegen (unreachable in practice). (`tests/cases/types/never-value-*.vl`, `never-narrowing-legit.vl`)
- **A14 fix: honest messages for degenerate self-referential aliases** — a pure self-cycle `type D = D` no longer claims `opaque/recursive type aliases are not yet supported` (which implied a future feature); it reports `Type \`D\` is defined in terms of itself with no base type, so it can never have a value` — it has no structural base, so nothing can inhabit it. A self-referential intersection `type E = A & E` (same class: a definition in terms of itself that collapses to `never`) reports `Type \`E\` refers to itself inside an intersection (\`A & E\`), so it reduces to \`never\``, ranged on the `A & E` expression. LEGITIMATE recursive aliases (`type List = { rest: List[] }`, mutual recursion) are unaffected — they resolve to a real structural body and never hit the degenerate-cycle path. (`tests/cases/types/self-alias-still-clean.vl`, `never-value-self-intersection.vl`, `recursive-array-element.vl`, `mutual-recursive-type.vl`)
- **B17 lint: empty-intersection warning** — a `type X = A & B` (or a binding so annotated) whose two CONCRETE operands share no values folds to `never`; the declaration is flagged with a `warning` (`code: empty-intersection`, `source: vital`) — `Intersection \`A & B\` is empty (\`never\`) — an object and a string literal, have no common values` — ranged on the `A & B` intersection expression (not the alias name), matching the never-value error. A self-referential `type E = A & E` instead warns `Type \`E\` refers to itself inside an intersection (\`A & E\`), so it reduces to \`never\``. Scoped to concrete operands: a generic `T & U` (empty only for some instantiation) is NOT flagged. (`tests/cases/lint/empty-intersection.vl`, `generic-intersection-no-warn.vl`)
- **A4 Negation surface syntax** — `!A` and `x !is T` (Kotlin-style negated guard); `subtractType` narrowing algebra. → `DECISIONS.md`
- **A5 Flow narrowing (broad)** — nullness, union-member, post-guard clauses, `&&`/`||` chains, De Morgan, else-of-else-if, literal discrimination, `?.`/`??`. (→ `docs/narrowing.md`)
- **A6 `is` operator + tagged unions** — `x is T` discriminates a value union at runtime (niche/value-kind/boxed encodings, global tag registry, `coerceUnion`). (→ `docs/unions.md`)
- **A6b Proof-carrying narrowing (degenerate)** — a body that is exactly `return <predicate-on-a-param>` is an inferred guard; guard-clause `if absent(v) { return }` supported.

- **A10 Parametric types / generics** — Stage 1 (function type params), Stage 2 (array element inference), Stage 3 (generic `type` aliases incl. nested/array); `map`/`filter` build-side generics over growable `T[]`. (`tests/cases/generics/`)
- **A-infer-null `let x = null` as a nullable hole** — an un-annotated `let x = null` is no longer pinned to the exact `null` type (which made `let x = null; x = 5` error `expected null, got 5`); `null` is now hole-bearing like `[]`. The binding is `T | null` where `T` is a fresh shared inference hole (`Nullable<Infer<Unknown>>`), and a later constraint — a reassignment `x = <expr>`, a `T | null` param, an annotated assignment — FILLS the hole's `T` in place via the same `constrainElementHole` machinery the empty array uses (`x = 5` ⇒ `x: i32 | null`). This is NOT a pin violation: VL still pins COMPLETE types (`let x = 7; x = "foo"` errors — `7` is complete, `null` is not), and once the hole is filled the cell behaves as that complete type (`let x = null; x = 5; x = "foo"` errors). The initializer contributes the `| null`, which is sticky (it reflects that `x` really was null) EXCEPT flow-narrowing strips it on a path where `x` is provably non-null: a straight-line `x = <non-null>` narrows `x` to its non-null base for the rest of the block (no null tax — `let x = null; x = 5; x + 1` runs), threaded through the shared `postGuardNarrowings` entry point so both the type checker's scope overlay and codegen's narrowed overlay get it; an assignment inside an `if` branch stays in that branch, so the conditional-assign idiom (`let x = null; if c { x = f() }`) still requires a null check across the branch. An UNCONSTRAINED `let x = null` (never reassigned/used to pin `T`) resolves to the plain `null` type — a lone `null` IS fully determined, so the inference floor (`reportUninferredBinding`) does NOT raise "cannot infer" for it (distinct from an empty `[]`, whose element is genuinely unknown). A `const x = null` (immutable) keeps the exact `null` type, and an explicit `let x: T | null = null` is unchanged. Also fixes a latent bug exposed here: typing an assignment EXPRESSION (a block-tail `{ x = v }`) whose target is any `T | null` cell (`let n: i32 | null` too) no longer spuriously fails the scalar `=` operator-method lookup. (`tests/cases/types/infer-null-*.vl`)
- **A-infer-empty Usage-driven element inference for empty `[]`** — an un-annotated empty array literal now infers its element type from DOWNSTREAM usage, exactly like a generic resolves its type arg from arguments: `const xs = []; xs.push(1)` infers `xs: i32[]` (and `xs.push("a")` → `string[]`), as does passing `xs` to a `T[]` param, an annotated assignment (`const ys: i32[] = xs`), a `T[]`-returning tail, or an index-set (`xs[i] = v`). The constraint is threaded back to the binding's shared element hole (`constrainElementHole` for `push`; the existing index-signature unify in `ensureType` for param/return/assignment), then the hole is collapsed in place so codegen sees a normal typed array. The "cannot infer — annotate" diagnostic (the robustness floor) is now deferred to scope-close, so it fires ONLY for a genuinely-unconstrained empty (no resolving use). (`tests/cases/arrays/infer-empty-*.vl`, `error-empty-uninferred.vl`)
- **A-infer-map-set Usage-driven key/value inference for `Map()` / `Set()`** — the `Map()`/`Set()` analogue of `A-infer-empty`. An un-annotated `const m = Map()` now infers its key/value type from a downstream `m.set(k, v)` (`const m = Map(); m.set("a", 1)` infers `m: {[string]: i32}`), and `const s = Set(); s.add("x")` infers `s: Set<string>`. The `Map()`/`Set()` constructor binds a bare `Infer<Unknown, mapCtor>` hole (the same object the binding holds); the first `.set`/`.add` MATERIALISES that hole IN PLACE (`constrainMapHole`) into the concrete string-keyed index-signature object — `{[string]: V}` for a Map, `{[string]: boolean}` for a Set — byte-for-byte identical to what an annotated `const m: {[string]: i32} = Map()` produces, so `.get`/`.has`/`.length`/`.keys()`/`.values()`/iteration/codegen all work unchanged. Every later `.set`/`.add` then type-checks STRICTLY against the now-pinned key/value, so a conflicting `m.set("b", "x")` after `m.set("a", 1)` (resp. `s.add(1)` after `s.add("x")`) is a clean type error, never a crash. Map/Set keys must be `string` (the only hash key the rep supports today); an inferred i32 key is rejected with the same message the annotated `{[i32]: …}` path uses. The "cannot infer — annotate" floor stays deferred to scope-close, firing ONLY for a genuinely-unconstrained `Map()`/`Set()` (no `.set`/`.add`). (`tests/cases/maps/infer-from-set.vl`, `maps/error-infer-conflict.vl`, `maps/error-uninferred.vl`, `sets/infer-from-add.vl`, `sets/error-infer-conflict.vl`)
- **A-infer-return-join Multi-return inference JOINS all returns into a union/nullable (and lowers it), not first-wins** — an un-annotated function whose `return`s disagree now infers the FLATTENED union/nullable of every return, not just the first one observed, AND the emitter lowers it end-to-end. Before, `function f(b) { if b { return 5 }; return null }` inferred `i32` (the first return), silently dropping the `null` branch — so a caller's `r != null` narrow was mis-typed (`cannot compare i32 with null`) and `function g(b) { if b { return 5 }; return "hi" }` was WRONGLY accepted as `i32`, then crashed codegen when the string flowed through an i32 result. **Inference:** the two return accumulators (`noteInferredRet`, and `noteNaturalRet` for the redundant-return rule) now fold each return through `joinTys` (which mirrors the host's `flattenType(Union(...))` — assignable-collapse, dedup, nullability folds to a `?` wrapper), so the caller sees the true `i32 | null` / `i32 | string`. **Codegen:** a VALUE union / nullable-scalar inferred return (`isValueUnionName` — ≥2 non-null value atoms, or one numeric scalar + `null`) now lowers through the SAME machinery as an annotation, driven from the checker's exported inferred name: `collectU` registers it (`registerValueUnionName` interns the `uBoxIdx` box + value-box reps even when an inferred return is the program's only value union), `buildFnMap` sets `fRetUni` (the result valtype), the return site boxes via `emitUnionCoerce`, and the caller's `unionNameOfExpr` resolves the inferred name so an `is`/`!= null` narrow unboxes correctly — all with NO annotation and NO change to the AST (`fmt`-invisible). Verified: 3-member unions (`i32 | string | boolean`), `f64 | null`, and an inferred union flowing into an annotated `i32 | string` param across functions. A ROBUSTNESS FLOOR remains only for the union shapes the value box can't yet hold — a union of STRUCT refs (variant-boxed rep) and the `boolean | null` / `string | null` niches — reported as `cannot infer a {union,nullable} return type for 'f' — annotate the return type` (mirroring the empty-`[]` floor) instead of a cryptic codegen crash; lowering those un-annotated is the remaining follow-up (`docs/vl-tech-debt.md`). Native fixpoint holds byte-for-byte; the compiler's own inference also shifted +20 bytes where a multi-return now joins to the common structural SUPERTYPE rather than the first branch — a strict correctness gain. (`tests/cases/functions/infer-{union,nullable}-return-codegen.vl`, `infer-struct-union-return-needs-annotation.vl`, `annotated-{union,nullable}-return.vl`)
- **A-nullable-bool `boolean | null` niche value type, end to end** — the `boolean | null` nullable rides a NICHE in the i32 cell (0 false / 1 true / 2 = the `null` sentinel — no box, no ref), and it now works as a first-class value type across param / let-assignment / return / call, not just in the narrow `.pop()` / optional-chain contexts it was stuck in. A `null` flowing into any of those positions emits the sentinel `2` (`pendingNulBool` seeded at the call-arg boundary `cParamNulBool`, the `x = null` assignment, and the return site `retNulBool`); a boolean stays `0`/`1`; the wasm result/param/local valtype is plain i32 (`vtKindOfType` → 0). The caller side resolves niche-ness through calls and un-annotated bindings (`exprNulBool` now recognizes a `: boolean | null` call and a `const r = f()` bound to one, via `letNulBool`), so `f(x) != null` is the sentinel compare (`== 2`) and the narrowed `if r != null { … r … }` reads the boolean. Before, every one of these failed `emitProgram: bare null needs a struct-typed context` (or rejected the param outright). The inferred (un-annotated) `boolean | null` return still hits the robustness floor (`annotate the return type` — the annotation now lowers); `??` over a niche is a follow-up. First of the nullable-value-type arc (`string | null`, `??`-over-niche, `if/else`-null-tail to follow — `docs/vl-tech-debt.md`). (`tests/cases/functions/nullable-bool-{return,param-let}.vl`)
- **A-nullable-string `string | null` value type, end to end** — `string | null` is the nullable-REF value type: `(ref null $aTypeIdx)` over the raw string array (kind 16), so `null` is `ref.null` and `!= null` is `ref.is_null` — distinct from the `boolean | null` i32 sentinel (PR1). It now works as a first-class value type across param / let-assignment / return / call / module-global, where before every position failed `bare null needs a struct-typed context` or rejected the param. A `null` emits `ref.null $aTypeIdx` (`pendingNulString` seeded at the call-arg boundary `cParamNulString`, the `x = null` assignment, the return site `retNulString`, and the let/global init); the valtype is `(ref null $aTypeIdx)` (`vtKindOfType` → 16, added to BOTH `fbValtype` and the local/global cell kinds — the original bug was a kind-16 case put in `fbValtypeNullable` instead of `fbValtype`, so the param ABI fell through to i32). The caller resolves niche-ness through calls and un-annotated bindings (`exprNullableString`/`letNulString`/`fnRetNulStr`), so `f(x) != null` and a `const r = f()` global both narrow correctly. The narrowed string is then read AS a value — `.length` (via `array.len`, which accepts the nullable ref), `==`, `+` concat, and return-as-`string` — by recovering the non-null ref (`ref.as_non_null`) at every value read, while the `!= null` check reads the RAW nullable ref (`pendingRawNullRead`) so `ref.is_null` stays meaningful. Two general fixes fell out: `retNulRefFlag` excludes a `string` base (it is the array, never a struct, so `string | null` is not mis-claimed as a nullable struct), and `exprIsStrEq` skips a `null` operand (a `== null` is a null-check, not element equality). Second of the nullable-value-type arc; `??`-over-niche and the `if/else`-null-tail follow (`docs/vl-tech-debt.md`). (`tests/cases/functions/nullable-string-{return,param-global}.vl`)

- **A-nullable-if-tail `if`/`else`-expression tail with a `null` branch** — `function f(b): i32 | null { if b { 5 } else { null } }` (the implicit-return form of `return null`) now lowers, where before it failed `emitProgram: unsupported statement in body` even ANNOTATED. Two parts: a trailing `null` is now a tail value (`stmtIsTailValue` gained the `NullLit` case — it was omitted, so the `if` fell to the void-statement path that can't emit a bare `null`); and the three implicit-return tail paths (`emitFuncBody`'s trailing value, `emitStmtTail`, the `if`-tail arms) now route through a shared `emitReturnValue` helper extracted from the `RetStmt` emitter — so a tail `null` / nullable value gets the SAME box / sentinel / `ref.null` seeding as `return null`, per the function's return type. Works for all three nullable reps (i32|null box, string|null ref, boolean|null niche, incl. `else if` chains). The helper takes a `widenNum` flag: explicit `return e` keeps numeric (f32/f64/i64) literal widening, the tail passes `false` (a tail value's type already matches the return, and forcing the float-widen mis-converts an already-f64 closure `call_ref` the classifier doesn't recognize — surfaced by routing the tail through the shared path). - **A-missing-return Fall-through totality — implicit `null`, not a trap** — a function whose body can fall off the end without yielding a value implicitly produces `null` there. That is sound exactly when `null` is part of the return type, so the rule is assignability-based, not a blanket reject: a **non-nullable** value return with a reachable fall-through is an error (`function f(n: i32): i32 { if n > 0 { return 1 } }` — the TS `noImplicitReturns` case, which used to type-check and then TRAP at the wasm result-type fallthrough), while a **nullable** return lowers the fall-through AS that null. An un-annotated function widens its inferred return to `T | null` on fall-through (host byline: "types hidden by aggressive inference") — `function f(n: i32) { if n > 0 { return 1 } }` infers `i32 | null`, no annotation needed — and a declared `: T | null` may fall through too (previously a false-positive reject). The totality test (`nodeIsTotal`) is the dual of the definite-assignment `daBranchDiverges`: a block is total iff its last statement is — an explicit `return`, a `break`/`continue` (divergence), an `if`/`else` whose BOTH arms are total (`else if` chains recurse), a `while true { … }` (never falls through), or a bare trailing VALUE expression. It is computed while the body scope is still live (a discriminated param/local's type would otherwise be out of scope) and consumed once the return type is finalized. The emitter pads a live fall-through with the return rep's null via `emitNullForRet` (boolean-niche sentinel `2`, string-niche `ref.null $array`, nullable struct `ref.null $S`, value-union null-box) instead of the dead `unreachable` it still uses for a proven-dead non-nullable fall-through. DISCRIMINATING `is` / literal-equality chains stay DEFERRED to the exhaustiveness check. Box-able value types (`i32`/`i64`/`f64`/`f32` and unions) infer un-annotated; a NICHE (`boolean | null` / `string | null`) inferred return still floors with "annotate the return type" until A20 carries the niche seed. (`tests/cases/soundness/missing-return-{reject,inferred-nullable,nullable-ok,niche-floor,ok}.vl`)
- **A-definite-assign Definite assignment for uninitialized locals** — an uninitialized `let x` / `let x: T` now starts UNassigned and NON-null (NOT implicitly nullable); a READ where `x` is not provably written on every preceding path is a `` `x` is used before it is assigned `` error. This closes a silent-`0` soundness gap (`let x: i32; return x` used to compile and fabricate a `0`). The check reuses the parser's single-pass flow machinery — the same `is`-narrowing fork/join that scopes through `if`/`else`/`while` and lets a diverging branch drop out: a per-`Binding` "unassigned" bit is forked at each branch, joined at the merge (assigned only if written on every surviving path — a `return`/`break`/`continue` branch contributes nothing, reusing `divergesStatement`), and reset to the pre-loop state after a `while`/`for`/`for-in` body (zero-iteration). The declaration itself is fine; only the reads are gated, and the binding's type is unchanged (`let x: i32` stays `i32`, not `i32 | null`). Initialized `let x = …` and every `const` are assigned from the start and untouched. (`tests/cases/variables/definite-assign-*.vl`)
- **A11 Recursive structural types** — `type Tree = { value, left: Tree | null, … }` constructs/traverses/compiles (cycle-safe traversals + self-referential WasmGC rec-group). (`tests/cases/types/mutual-recursive-type.vl`) Recursion through an **array element type** (`type List = { value: i32, rest: List[] }`) now compiles too — the element struct, its growable-list `{backing,len,cap}` wrapper, and the backing array are discovered as one strongly-connected component and built in a single WasmGC rec group (forward-referencing each other), instead of unboundedly re-entering type lowering and overflowing the stack. (`tests/cases/types/recursive-array-element.vl`) Recursion through a **map value type** (`type Tree = { value: i32, children: { [string]: Tree } }`) now compiles as well — the map's hash struct and its `vals` backing array join the same rec group, so the value-type back-edge is threaded via forward refs instead of overflowing through `mapType`→`mapArrayType`→`refHeapType`. (`tests/cases/types/recursive-map-value.vl`)

- **A18 Order-independent module scope + TDZ** — top-level `let`/`const` VALUES are now hoisted (names + types) in a checker pre-pass, exactly as top-level FUNCTIONS already were, so a function body resolves a global declared LATER in the file. Initialization order is still enforced: each global joins the definite-assignment dead-zone at hoist time and is marked assigned only when its initializer is reached, so a read before initialization — straight-line top-level code, OR one global's initializer reading a later global — is a `` `x` is used before it is assigned `` (temporal-dead-zone) error, NOT a silent read of uninitialized memory. A function body runs at call time, after module init, so its reads are correctly exempt (its own definite-assignment context is cleared). An annotated global hoists its annotation type; an un-annotated one hoists a side-effect-free LITERAL peek (a non-literal initializer stays order-dependent — it carries no manifest type to hoist). A genuine top-level redeclaration is still an error. (`tests/cases/scope/`)
- **A19 Binding-group return inference (recursive cycles)** — return-type inference now resolves a mutually-recursive GROUP whose return is determinable, with NO annotations: `even`/`odd`, or a chain whose return passes *through* a recursive cycle but bottoms out in a concrete value. `A17`'s demand inference is single-pass — at a cycle it commits a provisional `void` and memoizes it, so a function whose return only becomes known AFTER the cycle resolves inferred `void` and then errored. `A19` adds a suppressed return-inference PRE-PASS (before the body-checking pass) that iterates the cycle-poisoned ("tentative") functions to a FIXPOINT — the structural-typing analogue of HM's unify-the-binding-group, iterating the return lattice instead of unifying (VL has subtyping, not unification). It's standard SCC / binding-group inference (cf. *Typing Haskell in Haskell*, Haskell Report §4.5), realized lazily through the existing demand stack — which already visits callees first, i.e. dependency order — so no explicit call graph is built. Genuine VOID recursion (no value return position) still infers `void`; a genuinely base-case-less value cycle (`ping`→`pong`→`ping`, no concrete basis) still reports `cannot infer a return type`. The pre-pass cost scales with un-annotated functions and actual cycles — negligible on annotated code (no measurable change on `vl check compiler/`). (`tests/cases/functions/infer-*.vl`)
- **A20 Elaborate inferred return types for codegen** — the emitter classifies an UN-annotated function's return by re-deriving it from the return EXPRESSION syntactically (`exprString`/`exprStruct`/…); that misses a return which is a plain VARIABLE (a `let`-bound, concatenation-built string), so the function is mis-classified and a caller's field access crashes codegen (`field access receiver is not a struct`) — a checker-accepts / emitter-rejects soundness gap (the type checker proves a clean `string`, the emitter can't lower it), exposed by inferring more returns. The checker now exports each un-annotated function's inferred concrete SCALAR/STRING return name in a side table the emitter consults, and the emitter sets the classifying flag directly from that inferred TYPE — matching the declared-annotation path. It is kept OUT of the AST (`fnRet` is untouched), so `fmt` never re-adds the annotation: inference stays invisible in the source. Conservative on purpose: only scalar/string returns (names that round-trip unambiguously) and only NON-generic functions (a `self`-style `TyVar` param has a per-call-site return, so its definition-time inferred type is a placeholder — `sumsq(self)` rendered as `string`); structural / array / union returns keep the existing expression-based classification. (`tests/cases/functions/infer-string-return-codegen.vl`)
